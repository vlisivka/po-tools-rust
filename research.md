# Дослідження: Підтримка коментарів в ШІ-командах

## Тема дослідження
Включення коментарів PO-повідомлень (перекладацьких, витягнутих, посилань та прапорців) у промпти для AI-команд (`translate`, `review`, `filter`) для покращення контексту та точності обробки.

## Мета
З'ясувати причину відсутності коментарів у поточній реалізації та запропонувати оптимальний спосіб їх передачі в AI-бекенд.

## Поточний стан та діагностика

### 1. Проблема в парсері
Аналіз коду `src/parser.rs` виявив, що метод `parse_messages_from_stream` (який використовується для читання цілих файлів) явно ігнорує рядки, що починаються з `#`:

```rust
// src/parser.rs:571
} else if !line.starts_with('#') {
    if !buf.is_empty() {
        buf += "\n";
    }
    buf += line;
}
```

Через це у об'єктах `PoMessage`, отриманих під час читання файлу, вектор `comments` завжди порожній, навіть якщо у вихідному файлі коментарі були присутні.

### 2. Відображення в командах
Хоча структура `PoMessage` має коректну реалізацію `Display`, яка друкує коментарі, AI-команди не бачать їх через вищезгадану проблему в парсері. В результаті AI отримує лише `msgid` та `msgstr`, втрачаючи важливий контекст (наприклад, інструкції для перекладача або посилання на місце в коді).

## Опис варіантів вирішення

### Варіант A: Виправлення парсера (Мінімальні зміни)

Можна просто дозволити парсеру збирати всі рядки (включаючи коментарі) у буфер перед викликом `parse_message_from_str`.

**Плюси:**
- ✅ Найпростіша реалізація.
- ✅ AI бачить коментарі у стандартному форматі Gettext PO.
- ✅ Зберігається цілісність даних при переробці повідомлень.

**Мінуси:**
- ❌ Коментарі змішуються з технічними полями всередині тегу `<message>`, що може збивати деякі дрібні моделі (хоча більшість великих моделей добре розуміють формат PO).

### Варіант B: Виділення коментарів в окремий тег контексту

Окрім виправлення парсера, можна змінити формування промпту, щоб виносити коментарі в окремий блок (наприклад, `<comments>`).

**Приклад промпту:**
```xml
<comments>
#: src/main.rs:123
#. Translators: This is a greeting
</comments>
<message>
msgid "Hello"
msgstr ""
</message>
```

**Плюси:**
- ✅ Чітке розділення контексту та об'єкта обробки.
- ✅ Полегшує AI розуміння того, що коментарі — це допоміжна інформація.

**Мінуси:**
- ❌ Вимагає складнішої маніпуляції рядками в усіх AI-командах.

### Варіант C: Фільтрація типів коментарів

Не всі коментарі однаково корисні для AI. Посилання `#: ` зазвичай лише створюють шум. Можна передавати тільки перекладацькі коментарі `# ` та витягнуті `#.` .

**Плюси:**
- ✅ Зменшення кількості токенів.
- ✅ Фокусування AI на найважливішій інформації.

**Мінуси:**
- ❌ Потребує додаткової логіки фільтрації в коді.
- ❌ Можна випадково видалити корисні прапорці (наприклад, `fuzzy`).

## Рекомендації

Рекомендується реалізувати **Варіант A** в поєднанні з невеликим покращенням парсера.

**Обґрунтування:**
1. Коректний парсинг коментарів є базовою вимогою для інструменту роботи з PO-файлами. Це виправить не тільки AI-команди, а й потенційні проблеми з іншими командами (наприклад, `sort` або `merge` тепер будуть зберігати коментарі).
2. Формат PO є стандартним, і сучасні LLM (Gemma 3, GPT-4, Claude) чудово з ним працюють, сприймаючи коментарі як контекст до `msgid`.

**Кроки реалізації:**
1. У `src/parser.rs` змінити `parse_messages_from_stream`, щоб вона додавала коментарі до буфера.
2. Переконатися, що `parse_message_from_str` коректно обробляє коментарі на початку блоку.
3. Додати unit-тести для перевірки збереження коментарів при парсингу списку повідомлень.

## Вплив на проєкт

- **Файли**: `src/parser.rs`.
- **Поведінка**: AI-команди почнуть автоматично отримувати коментарі у тегу `<message>`, оскільки там використовується `Display` для `PoMessage`.
- **Супутні наслідки**: Всі команди виводу (наприклад, `sort`, `erase`, `merge`) тепер будуть коректно зберігати та друкувати коментарі, що є значним кроком до повної підтримки стандарту PO.
