## 1. Проблеми архітектури

### 1.1. Enum `PoMessage` — комбінаторний вибух варіантів

**Файл:** `src/parser.rs:10-34`

**Проблема:** Enum `PoMessage` має 5 варіантів, де `Regular`/`RegularWithContext` та `Plural`/`PluralWithContext` — це, по суті, одні й ті самі типи з опціональним `msgctxt`. Це призводить до:
- Дублювання коду у _кожній_ команді, де треба матчити 4-5 варіантів окремо.
- Комбінаторного вибуху у `with_key()` (рядки 72-158), де є 12+ гілок.
- Копіювання логіки в `command_print_translated.rs`, `command_print_untranslated.rs`, `command_print_with_word.rs`, `command_check_symbols.rs` тощо.

**Рекомендація:** Рефакторити `PoMessage` на структуру з опціональним `msgctxt`:
```rust
pub struct PoMessage {
    pub msgctxt: Option<String>,
    pub msgid: String,
    pub msgid_plural: Option<String>,
    pub msgstr: SmallVec<String>,  // Regular = vec з 1 елементом, Plural = vec з N елементів
}
```


**Пріоритет:** Високий. Це найбільше джерело дублювання коду в проєкті.

**Вплив на проєкт:** Всі 18 файлів-команд, `parser.rs`, `localization.rs`.

### 2.3. Неефективне використання `.replace("{}", ...)` для форматування

**Файли:** Множина файлів (`main.rs`, `command_translate_and_print.rs`, `command_compare_files_and_print.rs` тощо).

**Проблема:** Локалізовані рядки форматуються через `.replace("{}", value)`, але коли в рядку є кілька `{}`, кожний `.replace()` замінює **перше** входження. Це працює коректно лише випадково, через ланцюжок викликів. Приклади:
```rust
// command_translate_and_print.rs:93-95
tr!("Loaded {} messages from \"{}\" file with translation memory.")
    .replace("{}", &msgs.len().to_string())
    .replace("{}", tm_file)
```
Це працює, бо перший `replace` замінює перший `{}`, другий — другий. Але це крихкий підхід — якщо порядок `.replace()` зміниться, результат буде неправильним.

**Рекомендація:** Використовувати іменовані плейсхолдери `{count}`, `{file}` замість `{}` або format!().

**Пріоритет:** Середній. Поточний підхід працює, але крихкий та неінтуїтивний.

## 7. Структурні покращення

### 7.1. Відсутність підтримки коментарів у виводі PO-повідомлень

**Файл:** `src/parser.rs`

**Проблема:** Парсер зберігає коментарі PO-файлу (рядки `#`, `#:`, `#,` тощо), але при виводі повідомлень вони **втрачаються**. Це означає, що при обробці файлів зникають:
- Коментарі перекладачів (`#`)
- Посилання на вихідний код (`#:`)
- Прапорці (`#,` — fuzzy, c-format тощо)
- Попередній переклад (`#|`)

Це є принциповим обмеженням для інструменту роботи з PO-файлами.
